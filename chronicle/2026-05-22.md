## What was decided

- **Render's free tier is good enough for staging**, despite the ~30–60s cold-start. Cost > performance at this stage. Cold-start mitigation deferred.
- **Secrets-in-env, not secrets-in-file, for cloud hosts.** Render gets the service-account key as a pasted JSON string via `GOOGLE_SERVICE_ACCOUNT_JSON`. Avoids needing to mount secret files in the host.
- **Both hosts auto-deploy on push to `master`.** Trade-off accepted: faster iteration, occasional broken-master risk. No staging branch yet.

## Reusable insights
- **The "devDeps in production" gotcha** is the kind of thing every Node-on-Render deploy hits once and never again. Worth a `playbooks/deploying-a-node-ts-backend-to-render.md` entry. The fix: `npm install --include=dev && npm run build` — explicit because `NODE_ENV=production` makes npm skip devDeps, which silently includes the TypeScript compiler.
- **`CORS_ERR_INVALID_CHAR` from a trailing newline in an env var** is a sharp-edge worth capturing. The fix (split on `,`, trim each, filter empty) is general — belongs in a `core/` utility if more Execute AI solutions end up with CORS layers.
- **`vercel.json` SPA rewrite** is boilerplate-grade reusable. Belongs in a `templates/vercel-spa.json` somewhere in `shared/` or `tools/`.
- **Service-account ownership / Shared Drive** is THE Google Workspace integration gotcha. Every future Execute AI Google-stack solution will hit it. The CLAUDE.md note is good — but a `playbooks/google-workspace-service-accounts.md` would prevent the next rediscovery.
